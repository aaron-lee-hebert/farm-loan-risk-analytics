# Flow ITAM

**Flow ITAM** is a development project focused on building a simple, scalable IT Asset Management (ITAM) system from the ground up using modern Python tooling.

This project is intentionally designed as a **learning-first, architecture-focused system**, with the long-term potential to evolve into a production-ready API and SaaS offering.

---

## 🎯 Purpose

The goal of Flow ITAM is to:

* Deepen understanding of **backend system design**
* Practice **clean architecture principles** in Python
* Gain hands-on experience with **explicit data access (SQLAlchemy Core)**
* Build a system that can evolve from a **CLI tool → Web API → full application**
* Explore real-world concerns like:

  * Data modeling
  * Performance
  * Maintainability
  * Scalability

This is not just a coding exercise—it is an intentional effort to grow toward **high-impact, system-level engineering**.

* Track assets (laptops, monitors, etc.)

## 🏗️ Architecture

This project follows a layered approach inspired by Clean Architecture, implemented as Python packages under `src/flow_itam/`:

* **`domain/`**
  Core business entities (Pydantic models) and rules — no I/O, no DB, no HTTP

* **`application/`**
  Use cases and orchestration logic (services); depends on `domain/` only

* **`infrastructure/`**
  Data access (SQL Server via SQLAlchemy Core + `pyodbc`), config, external integrations

* **`cli/`**
  Initial interface for interacting with the system (Click commands)

* *(Planned)* **`api/`**
  FastAPI layer that reuses the `application/` services

The goal is to keep business logic independent from delivery mechanisms, allowing the system to evolve without major rewrites. Python doesn't enforce layer boundaries the way C# project references do — discipline (and optionally `import-linter`) keeps the dependency arrows pointing inward.

---

## ⚙️ Technology Stack

* **Python 3.12+**
  Core runtime

* **SQLAlchemy Core**
  Expression-builder for explicit, readable SQL — no ORM session, no lazy-loading surprises

* **Alembic**
  Schema migrations, autogenerated from SQLAlchemy `MetaData`

* **Pydantic v2**
  Domain models with built-in validation; doubles as request/response schemas in the API phase

* **Click**
  CLI framework — decorator-based command groups

* **SQL Server**
  Relational database for structured data and reporting (via `pyodbc` + ODBC Driver 18)

* *(Planned)* **FastAPI**
  For external access and integration; pairs natively with Pydantic models

## 🧠 Development Approach

This project emphasizes:

### 1. Iterative Development

Small, incremental changes over time rather than large rewrites.

### 2. Learning by Building

Each feature is an opportunity to explore:

* Better design decisions
* Tradeoffs in data access
* Real-world constraints

### 3. Simplicity First

Avoid overengineering. Focus on:

* Clear code
* Understandable structure
* Maintainable patterns

### 4. Separation of Concerns

* Business logic is isolated from infrastructure and UI
* The CLI is treated as a temporary interface layer
* Repositories return domain models, not raw rows

### 5. Performance Awareness

Using SQLAlchemy Core (not the ORM) to better understand:

* SQL queries — you write them, you read them
* Execution performance via `SET STATISTICS IO, TIME ON` and actual execution plans
* Data access patterns
